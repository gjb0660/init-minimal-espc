# Spec–Code Alignment

## Overview

在任何由人类或 Agent（尤其是 LLM）参与的软件系统中，**spec 与 code 的偏离（drift）是必然现象，而非异常情况**。

这种偏离并不意味着系统失败，而是反映了：

- 认知与实现之间的差距
- 信息不完备条件下的渐进式收敛过程
- 多轮决策与局部修复带来的结构演化

因此，问题的核心不在于“如何避免偏离”，而在于：

> **如何在允许短暂不一致的前提下，实现可控、可收敛的一致性恢复机制**

---

## Why Drift Happens

### 1. 不完备认知（Incomplete Understanding）

无论是人类还是 Agent，在初始阶段都无法完全掌握：

- 真实需求边界
- 隐含约束
- 系统上下文

因此：

- spec 可能过度理想化或不完整
- code 可能基于局部理解进行实现

这种偏差是信息逐步揭示的自然结果。

---

### 2. 多轮迭代与局部修复（Iterative Local Fixes）

在实际开发中，系统通常经历：

- bug 修复
- 局部优化
- 临时补丁

这些操作往往：

- 没有回溯到整体 spec
- 只针对局部问题

从而导致：

> **实现逐渐偏离最初的结构表达**

---

### 3. Agent 的上下文限制（Context Window Constraints）

在 Agent 驱动开发中：

- 每一轮执行只能看到有限上下文
- 无法保证全局一致性

结果是：

- 每一轮可能“局部合理”
- 多轮叠加后出现全局偏差

---

### 4. 幻觉与推断（Hallucination and Inference）

LLM 在缺乏明确信息时，会：

- 自动补全逻辑
- 推断“合理行为”

这会导致：

- code 实现出 spec 未定义的行为
- 或错误理解 spec 的语义

---

### 5. 现实反馈（Reality Feedback）

在实现过程中，开发者或 Agent 可能发现：

- 原 spec 不可行
- 某些假设不成立
- 新需求被引入

这会产生一种更复杂的情况：

> **spec 本身也可能变得不再正确**

---

## Types of Drift

在实践中，spec–code 偏离通常可以归纳为几种基本形态：

### 1. Spec Overstates Code

spec 声称某种行为存在，但实现未覆盖。

典型表现：

- 功能缺失
- 验收条件未满足

---

### 2. Code Exceeds Spec

实现提供了 spec 未定义的能力。

典型表现：

- 隐式扩展
- 行为未被约束或文档化

---

### 3. Ownership Drift

行为存在，但归属层错误。

例如：

- 应属于某一功能层的逻辑，被实现到更底层或更高层

---

### 4. Foundational Contradiction

不同层之间的定义发生冲突，例如：

- 不同模块对同一行为有不同定义
- 约束与实现互相矛盾

这是最严重的一类偏差。

---

### 5. Documentation Drift

仅表现为描述层不一致，不影响实际行为。

---

## Core Principle

在面对偏离时，一个核心原则是：

> **Spec 表达当前成立的结构，而不是历史实现的记录。**

这意味着：

- spec 不应被动“追认”实现
- code 不应被视为默认更真实的来源

换句话说：

> **偏差是执行问题，而不是结构事实。**

---

## A Minimal Alignment Model

可以用一个最小模型来理解系统：

| 层级 | 含义 |
| -- | -- |
| Spec | 当前成立的结构（Truth） |
| Code | 对结构的实现（Implementation） |
| Drift | 两者之间的差异（Error） |
| Todo / Report | 显式记录的偏差（Delta） |

关键不变量：

> **结构层（Spec）与偏差层（Drift）必须分离**

---

## Alignment Strategy

一个有效的对齐过程，必须包含三个基本步骤：

### 1. Detect（识别偏差）

- 对比结构声明与实际行为
- 仅记录可验证的差异

---

### 2. Classify（分类偏差）

每个偏差必须回答：

- 偏差类型是什么？
- 责任属于哪一层？
- 影响范围如何？

---

### 3. Decide（决策路径）

每个偏差只能进入三种路径之一：

#### Repair Code

当结构仍然成立时：

- 修复实现，使其符合结构

---

#### Update Spec

当新的、可验证事实推翻原结构时：

- 先更新结构表达
- 再调整实现

---

#### Escalate

当存在基础冲突或无法判定时：

- 暂停局部修复
- 上升到更高层决策

---

## The Role of Explicit Drift Recording

在多轮迭代中，无法保证一次性收敛，因此需要：

> **一个显式的偏差记录机制**

它的作用是：

- 累积未解决问题
- 支持多轮收敛
- 降低上下文丢失风险

但它必须满足：

- 不与结构层混合
- 不成为新的真源
- 不替代决策过程

---

## Common Failure Modes

### 1. Spec 追认实现

将当前实现直接写入结构定义。

结果：

- 结构失去约束力
- 系统退化为“代码即真理”

---

### 2. 实现优先主义

默认认为：

> “代码已经这样了，所以它是对的”

结果：

- 错误被固化
- 设计逐渐失控

---

### 3. 隐式词表扩展

在分类或决策时随意发明新概念。

结果：

- 语义不稳定
- 判断不可复现

---

### 4. 混合多层职责

将：

- 结构定义
- 偏差记录
- 执行计划

混在同一层中。

结果：

- 边界模糊
- 系统复杂度迅速上升

---

## Design Implications

### 1. 允许短暂不一致

系统不需要始终强一致，但必须：

- 显式化偏差
- 控制其范围

---

### 2. 强制显式决策

每个偏差必须被明确处理：

- 修实现
- 改结构
- 或上升决策

---

### 3. 保持结构层纯净

结构表达必须始终：

- 简洁
- 当前有效
- 不包含历史噪音

---

### 4. 限制表达空间

通过有限的分类与决策集合：

- 限制系统复杂度
- 提高一致性

---

## Long-Term Insight

Spec–Code Alignment 本质上不是一个“修复问题”，而是一个：

> **持续收敛系统（Continuous Convergence System）**

它的核心能力是：

- 在不完美信息下前进
- 在偏差中逐步逼近一致性
- 在演化中保持结构稳定

最终目标不是“没有偏差”，而是：

> **任何偏差都可以被发现、解释，并以最小成本收敛。**
